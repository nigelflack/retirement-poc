# v0.14 ‚Äî Full input editor

_Status: Complete_

---

## Goal

Expose all JSON-editable fields in the UI. Currently `incomeStreams`, `capitalEvents`, and multi-step schedules are passthrough-only ‚Äî editable in JSON but invisible in the UI. This iteration makes every field editable directly.

---

## What's new

### 1. Assets screen (Step 2) ‚Äî per-person Contributions and Income streams sections

The existing tabbed screen gains two collapsible sections per person:

**Contributions** ‚Äî replaces the simple "reduces to ¬£X in last N years" step-change in ScenarioScreen. Full list editor: any number of rows, each with:
- `fromYearsFromToday` (integer, 0 = today)
- `monthlyAmount` (¬£)
- `label` (optional text)

When the list is empty, the adapter falls back to summing per-account `monthlyContribution` (existing behaviour, unchanged). When populated, it overrides per-account amounts.

**Income streams** ‚Äî was not editable at all. Full list editor: any number of rows, each with:
- `fromYearsFromToday` (integer)
- `toYearsFromToday` (integer, optional ‚Äî leave blank = run to end of simulation)
- `monthlyAmount` (¬£)
- `label` (optional text)

Both sections initialise from `people` prop values when present (e.g. after a file load then "Edit accounts").

### 2. Assets screen ‚Äî Household tab

A new `Household` tab is added alongside the person tabs. Contains:

**Scenario label** ‚Äî single text input. Stored as top-level `label` in the saved JSON.

**Capital events** ‚Äî was not editable at all. Full list editor: any number of rows, each with:
- `yearsFromToday` (integer)
- `amount` (¬£, signed ‚Äî positive = inflow, negative = outflow)
- `label` (optional text)

### 3. ScenarioScreen ‚Äî simplification and spending schedule upgrade

- **Remove** the per-person contribution step-change section and all associated state (`contribReducesTo`, `contribLastYears`, `contribStepOpen`). Per-person contributions are now set via Assets.
- **Upgrade** the household income step-change ‚Üí full **spending schedule** list editor (same shape as contribution schedule but uses `fromYearsFromRetirement`). Any number of rows, each with `fromYearsFromRetirement`, `monthlyAmount`, optional `label`. When empty, flat `monthlySpendingTarget` is used (existing fallback).
- `buildPayload`: reads `contributionSchedule` and `incomeStreams` directly from the `people` prop (set by Assets) rather than building from step-change state.
- `capitalEvents` and `label` come from the new household state passed through from Assets.
- `handleSave` and `handleFileChange` updated to round-trip all new fields.

### 4. State threading

ScenarioScreen needs household-level fields (`capitalEvents`, `label`, `spendingSchedule`) and these are now editable. The existing `loadedCapitalEvents` state is replaced by `capitalEvents` (editable list). `scenarioLabel` and `spendingSchedule` are new state items.

When a file is loaded in ScenarioScreen (`handleFileChange`):
- `onPeopleLoad(data.people)` ‚Äî carries contribution schedules and income streams up to App state (existing mechanism)
- Household fields set directly in ScenarioScreen state: `setCapitalEvents(...)`, `setScenarioLabel(...)`, `setSpendingSchedule(...)`

## Addendum ‚Äî `retirementAge` in JSON

**Change**: `retirementAge` is now saved per person in the JSON and restored on load.

- **`handleSave`** in ScenarioScreen embeds the current retirement age spinner values into each person object: `{ ...person, retirementAge: retirementAges[person.name] }`.
- **`handleFileChange`** restores retirement ages from `data.people[].retirementAge` (falls back to `currentAge + 10`, max 65 if absent).
- **`defaultRetirementAges`** initialisation also reads `person.retirementAge` if present, so a scenario loaded via ScenariosPage or via `location.state` will use the saved age.
- Sample files updated: Nigel 62, Mimi 62 (nigel-mimi.json); Alice 65, Bob 65 (bob-alice.json). CLI sample file updated to match.
- No server changes.
