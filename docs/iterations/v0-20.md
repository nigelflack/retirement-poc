# v0.20 — Tax Approximation + Net Worth Split + Taper Warning

**Status:** complete

---

## Goal

Increase model realism without overcomplicating inputs by adding:

1. A basic UK tax approximation (good directional accuracy; target within a few percent for typical PAYE-style cases).
2. Net-worth output split between liquid and non-liquid assets.
3. Pension annual allowance taper warning (warning only, no hard enforcement yet).

---

## Scope

### In scope

- Add a taxability marker to income items and treat untaxed/taxed flows differently in adapter resolution.
- Add an approximate UK tax calculation layer for taxable income in each year.
- Convert spending funding pressure into gross draw requirement when taxable pension draw is used (approximate gross-up).
- Return and display liquid vs non-liquid net-worth trajectories/summary in API response/UI.
- Add taper warning detection and surface warnings in response/UI.

### Out of scope / deferred

- Full HMRC-accurate tax engine (all edge cases, NI class detail, Scottish bands, carry-forward optimization).
- Hard cap/enforcement of tapered annual allowance contributions.
- Employer contribution + carry-forward optimization in solve endpoints.
- Tax treatment for dividends/capital gains as separate regimes.

---

## Design decisions

### Tax model level: approximation first

Implement a simple, transparent tax approximation suitable for planning:

- taxable vs non-taxable income flag per income item
- annual UK-style income tax bands (configurable)
- optional personal allowance taper approximation at high incomes

This release aims for planning-quality estimates, not payroll-grade accuracy.

### Preserve existing scenario compatibility

If a schedule item has no taxability field, default behavior remains backward-compatible using sensible defaults (income treated as net unless marked taxable).

### Warning-first taper support

If inferred pension contributions appear above tapered allowance thresholds, return warnings in response with per-year visibility in debug mode. Do not block simulation and do not auto-adjust contributions in v0.20.

### Net worth as first-class output

Expose both liquid and non-liquid values so late-life outcomes are not misread as total insolvency when home/property equity exists.

---

## API/data shape changes (planned)

### Request additions

- `incomeSchedule[]` optional metadata:
  - `taxable: boolean` (default `false` for backward compatibility)
  - `taxCategory?: 'employment' | 'pension' | 'rental' | 'other'` (optional, for future extension)

### Response additions

- `warnings: string[]` (run-level summary; includes taper warnings)
- `netWorthPercentiles` object with per-year liquid/nonLiquid/total real and nominal percentiles
- optional debug tax breakdown by year when `debug=true`:
  - `resolvedYears[y].tax` (estimated tax burden used in that year)
- optional per-year warning details when `debug=true`:
  - `resolvedYears[y].warnings[]` (includes taper warning entries for that year)

## Acceptance checks

1. Existing scenarios without `taxable` fields still run successfully.
2. Marking an income item as taxable reduces projected available spend capacity versus non-taxable baseline.
3. When taxable draw is required, debug output includes a non-zero tax estimate for relevant years.
4. Response includes net-worth split (liquid, non-liquid, total) with percentile series.
5. UI shows liquid vs non-liquid context so users can distinguish spendable assets from total wealth.
6. If estimated pension contribution exceeds tapered threshold conditions, response includes at least one summary warning string and debug output includes warning entries on affected years.
7. Selected-year UI view shows taper warning text when the selected year contains taper warnings.
8. Simulation is not blocked by taper warning (warning-only behavior).
9. `npm run build` in `ui/` passes and server tests pass.
