# v0.22 — Per-Person Tax, Property + Mortgage (Including BTL), and Depreciating Assets

**Status:** complete

---

## Goal

Three calculation improvements:
1. Tax and NI calculated per-person (not pooled household) during work years — removes the biggest remaining in-work accuracy distortion.
2. Property modeling (main residence and BTL) — property value, mortgage liability, rental cashflow, and mortgage tax-credit handling for BTL.
3. Depreciating assets (car) — non-liquid asset with configurable depreciation, saleable via capital event, optional lease cost after disposal.

---

## Scope

### In scope

#### 1. Per-person tax and NI

- Income items gain an optional `owner` field (person id)
- Adapter computes tax and NI per person: groups taxable/employment income by owner, runs `calculateUKTax` and `calculateUKNI` separately for each person
- Combined household tax/NI injected as expense items as before
- Income items with no `owner` remain household-level and are pooled (backward compatible)
- State pension synthetic items inherit their person's id
- Marginal rate for relief-at-source is computed per person (highest earner's rate used if contribution is ambiguous across persons)

#### 2. Property + mortgage (main residence and BTL)

- New pot type: `"property"` already exists but gains first-class residence support via optional `mortgage` object per pot:
  - `outstandingBalance` — current mortgage balance
  - `mortgageType` — `"repayment"` (default) or `"interestOnly"`
  - `interestRate` — annual mortgage interest rate
  - `annualPayment` — total annual payment (principal + interest) for repayment mortgages
  - `overpaymentAnnual` or `overpaymentPct` — optional annual overpayment amount or fraction of outstanding balance
  - `monthlyRent` — optional gross monthly rent (BTL)
  - `monthlyExpenses` — optional monthly operating expenses (BTL: agency/maintenance/etc.)
- Mortgage + BTL behavior:
  - Each year the outstanding balance reduces by `annualPayment + overpayment − interest`
  - For `interestOnly`, outstanding balance does not amortize from regular payments (only via overpayment if configured)
  - When outstanding balance reaches zero, mortgage payments stop automatically (no manual `toYear` needed)
  - Mortgage payments are injected as expense items in the year array (replacing a manually added expense schedule entry)
  - For BTL, yearly rent and yearly expenses are derived from monthly values and inflated with the configured inflation assumptions
  - BTL taxable income contribution is modeled as `annualRent − annualExpenses − annualMortgagePayments` (pragmatic approximation)
  - Annual mortgage interest for BTL is tracked and a 20% tax credit is applied against total tax liability (`credit = 0.20 × interestPaid`, floored at zero tax)
- Net equity shown correctly: property value − outstanding balance contributes to non-liquid net worth
- On sale (capital event), net proceeds = sale amount − outstanding balance at that year
- Inflation behavior:
  - Property value follows existing property return assumptions (nominal)
  - BTL rent and BTL expenses inflate each year
  - Mortgage outstanding balance remains in nominal pounds and is additionally reported in real terms in debug output

#### 3. Depreciating assets (car)

- New pot type: `"depreciating"` — non-liquid asset that applies a negative return each year
- Pot definition gains optional `annualDepreciationPct` (default 0 — no depreciation)
- Engine applies `1 − annualDepreciationPct` as return factor per year (deterministic, no volatility)
- Treated as non-liquid for net-worth purposes (same as property)
- Saleable via standard capital event (year, amount, optional `haircut` fraction for quick-sale discount)
- Optional: add an expense schedule item for ongoing running costs or post-sale lease cost (user-managed, not auto-injected)
- Backward compatible — existing pots unaffected

### Out of scope / deferred

- Salary sacrifice, Scottish rates, pension carry-forward
- Retirement-phase per-person tax (pension drawdown, state pension tax treatment) — to follow in a later iteration
- Vehicle running cost auto-injection
- Partial pot liquidation (fraction of car value sold)
- Solve endpoint support for new fields

## Design decisions

### Per-person tax is adapter-only

The engine has no knowledge of persons. Per-person tax grouping is done in the adapter (`simulate.js`) before building the year array. Combined household tax/NI flows into the engine as before.

### Mortgage as first-class adapter concern

The adapter resolves the mortgage schedule year-by-year before passing to the engine. The engine receives a plain expense item for mortgage payment each year it is active; it has no knowledge that this is a mortgage. Net equity is tracked via a companion `mortgageBalance[potId]` array in the adapter, used to correctly report equity in debug output.

### BTL simplification approach

BTL is intentionally modeled with a pragmatic approximation suitable for planning, not full tax-accounting fidelity:
- tax base uses `rent − expenses − mortgage payments`
- an additional 20% mortgage-interest credit is applied to tax liability

This may under/overstate tax versus strict Section 24 treatment in edge cases, but remains within accepted planning tolerance for this project.

### Depreciating asset return model

Depreciation is modeled as a deterministic negative return applied in the engine's per-pot return step. The engine already applies per-pot return factors; the adapter sets `annualDepreciationRate` on the pot definition and the engine uses it instead of the stochastic model for `depreciating` type pots.

### Sale haircut

Capital events for depreciating assets support an optional `haircut` fraction (e.g. `0.1` = 10% below current pot value). If both `amount` and `haircut` are supplied, `haircut` takes priority (proceeds = `potValue × (1 − haircut)`). If neither, the capital event `amount` is used as normal.

## Acceptance checks

1. Existing scenarios without new fields run with identical behavior.
2. A scenario with two people having different income levels produces different per-person tax than a pooled calculation would.
3. An income item with no `owner` still contributes to household tax (backward compat).
4. A mortgage with `outstandingBalance`, `annualPayment`, and `interestRate` auto-stops generating expense items when the balance reaches zero.
5. Overpayment (fixed or percentage) reduces the outstanding balance faster and causes earlier payoff.
6. Net equity on sale via capital event deducts the outstanding balance at that year.
7. A property configured as BTL with `monthlyRent` and `monthlyExpenses` adds yearly inflating rent/expense cashflow lines.
8. BTL taxable income uses `annualRent − annualExpenses − annualMortgagePayments` and applies an additional `20% × interestPaid` tax credit.
9. BTL mortgage outstanding balance remains nominal while debug output also includes a real-terms balance view.
10. A `depreciating` pot loses value each year at `annualDepreciationPct` deterministically.
11. A capital event with `haircut` on a depreciating asset produces proceeds below current pot value.
12. Server tests pass and `npm run build` in `ui/` passes.
