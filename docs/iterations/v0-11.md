# v0.11 — Schedule-aware simulation engine + API overhaul

_Status: Complete_

---

## What's new

### 1. Rename endpoint: `/run` → `/simulate`

`POST /run` becomes `POST /simulate`. The name better reflects what it does and distinguishes it from the solve endpoints, which are searches over the simulation. `/solve/income` and `/solve/ages` are unchanged in path.

---

### 2. Replace `withdrawalRate` with `annualIncomeTarget`

The API has historically accepted `withdrawalRate` (a fraction of the pot at retirement) as the income input. This is a derived quantity, not a user concept — it required the UI to do a roundtrip approximation to convert monthly income to a rate. It also fundamentally cannot support an income that changes over time.

**`withdrawalRate` is removed from the request.** It is replaced by `annualIncomeTarget` — the desired total household annual income in today's money. This is the base case for a flat income through all drawdown years.

The response continues to include `withdrawalRate` as a **computed output field** (derived from `annualIncomeTarget / accumulationSnapshot.real.p50`) for informational display. It is never an input to any endpoint.

---

### 3. New optional schedule fields on `POST /simulate`

Three optional fields allow time-varying simulation inputs. If any field is absent, the adapter fills in the historical flat behaviour (see adapter section below).

#### `contributionSchedule` (per person, optional)

A step schedule of annual contribution amounts for that person, in today's money. Steps are defined by `fromYear` — years from today (i.e. `targetAge - person.currentAge`). Contributions are fixed at zero from `retirementYear` onward regardless of schedule.

```json
"contributionSchedule": [
  { "fromYear": 0, "annualAmount": 9600 },
  { "fromYear": 10, "annualAmount": 14400 }
]
```

If absent, the person's current total monthly contribution (`accounts[].monthlyContribution` summed × 12) is used flat until retirement.

#### `incomeSchedule` (household, optional)

A step schedule of total household annual income target, in today's money. Steps defined by `fromYear` — years from today relative to the reference person (earliest retiring). Entries before `householdRetirementYear` are ignored.

```json
"incomeSchedule": [
  { "fromYear": 22, "annualAmount": 40000 },
  { "fromYear": 35, "annualAmount": 28000 }
]
```

If absent, `annualIncomeTarget` is used flat for all drawdown years.

#### `capitalEvents` (household, optional)

Lump-sum capital additions or withdrawals at specific simulation years. `year` is years from today relative to the reference person. `amount` is signed: positive = inflow, negative = outflow. Applied to the household pot in the simulation year before the return factor is applied.

```json
"capitalEvents": [
  { "year": 18, "amount": 50000 },
  { "year": 24, "amount": -10000 }
]
```

If absent, no capital events occur.

---

### 4. Server-side adapter

A new function `resolveSchedules(input, totalYears)` is added in the route handler (`server/src/routes/simulate.js`) — **not** in the engine. It runs once per request, before `runFull` is called. It resolves the sparse schedule inputs into dense arrays of length `totalYears`:

- `contributionByYear[personIndex][year]` — annual contribution for that person in that year (0 from `retirementYear` onward)
- `incomeTargetByYear[year]` — annual income target for that year (0 during accumulation)
- `capitalEventsByYear[year]` — net signed capital event for that year (0 if none)

`totalYears` is computable from the input (earliest retirement year + drawdown years from `toAge`).

The adapter is the only place that handles the sparse → dense conversion. The engine never sees schedules.

---

### 5. Engine changes (`server/src/simulation/run.js`)

`runFull` signature changes. It no longer accepts `withdrawalRate`. It accepts the three pre-resolved dense arrays from the adapter.

**Accumulation phase** — per person per year:
```
pots[pi] += contributionByYear[pi][y]
pot (household) += capitalEventsByYear[y]   // applied after contributions, before return
```

**Drawdown phase** — per year:
```
inflatedTarget = incomeTargetByYear[y] * cumInflation
totalSP = sum of active state pensions * cumInflation
draw = max(0, inflatedTarget - totalSP)
if draw > pot → ruined
pot -= draw
pot += capitalEventsByYear[y]               // positive = inflow, negative = outflow
pot *= returnFactor
```

No other changes to the simulation logic.

---

### 6. `/solve/income` and `/solve/ages` updated

Both solve routes currently derive `withdrawalRate` internally from a p50 seed run. This is removed.

- `/solve/income` — accepts `annualIncomeTarget` is not needed as input; the binary search already operates over `monthlyIncome`. The internal seed run used to get p50 is kept (it's needed to bound the search), but it now calls `runFull` with a flat `incomeTargetByYear` array derived from a trial rate. The response drops `withdrawalRate` from output.
- `/solve/ages` — the internal `withdrawalRate` derivation (`(monthlyIncome × 12) / p50`) is removed. Instead, `annualIncomeTarget = monthlyIncome × 12` is passed directly as a flat `incomeTargetByYear` array.

Neither solve endpoint accepts `incomeSchedule`, `contributionSchedule`, or `capitalEvents` — solve operates on the flat income/contribution model only.

---

### 7. CLI updates

- All calls to `/run` change to `/simulate`.
- The withdrawal rate prompt in the standard run loop is replaced by a **monthly income target prompt** (`Monthly income target (£/mo)`). The CLI converts to annual (`× 12`) before constructing the payload.
- The `--solve income` and `--solve ages` modes already prompt for monthly income — no prompt change needed, but the internal payload construction is updated to use `annualIncomeTarget`.
- `formatter.py`: the `Withdrawal rate` display line is replaced by `Effective rate` computed from `withdrawalRate` in the response (which remains an output field).
- Updated scenario JSON files (e.g. `inputs/nigel-mimi.json`) gain an `annualIncomeTarget` field; `withdrawalRate` is removed from input files.

---

### 8. CLI `--debug` flag

A new `--debug` flag on the standard run mode (not solve) prints a full year-by-year table after the normal output. Columns:

```
Year  Age  Phase  Contribution  Income target  Capital event  p10 (real)  p50 (real)  p90 (real)
```

- Phase: `A` (accumulation) or `D` (drawdown), with `← retire` annotation on the retirement year
- Contribution: annual total for the household in that year (from `contributionByYear`)
- Income target: `annualIncomeTarget` for that year (today's money), `—` during accumulation
- Capital event: net amount for that year, `—` if zero
- Portfolio percentiles from `portfolioPercentiles.byAge[n].real`

This requires the server to return the resolved schedule arrays in debug mode. A new optional request field `debug: true` causes `/simulate` to include `resolvedSchedules` in the response (the three dense arrays). Normal requests never include them.

---

### 9. Contribution inflation adjustment (post-complete patch)

Contributions were being added to pots as raw nominal amounts, while income targets and all reported values are in today's money. This meant contribution purchasing power was silently eroding each year in the simulation.

**Fix:** the engine now multiplies each contribution by `cumInflation` before adding it to the pot:

```
pots[pi] += contributionByYear[pi][y] * cumInflation
```

This makes all values consistent: contributions defined in today's money are correctly inflated forward to the nominal year before being added to the nominal pot.

## API changes

### `POST /simulate` (was `POST /run`)

**Request — removed field:**
- `withdrawalRate` — removed

**Request — new required field:**
- `annualIncomeTarget` (number) — desired total household annual income in today's money

**Request — new optional fields:**
- `people[].contributionSchedule` — array of `{ fromYear, annualAmount }` step entries
- `incomeSchedule` — array of `{ fromYear, annualAmount }` step entries (household)
- `capitalEvents` — array of `{ year, amount }` (signed, household)
- `debug` (boolean, default false) — if true, response includes `resolvedSchedules`

**Response — changed:**
- `withdrawalRate` remains as a **computed output** field: `annualIncomeTarget / accumulationSnapshot.real.p50`

**Response — new (debug only):**
- `resolvedSchedules.contributionByYear` — 2D array `[personIndex][year]`
- `resolvedSchedules.incomeTargetByYear` — 1D array `[year]`
- `resolvedSchedules.capitalEventsByYear` — 1D array `[year]`

### `POST /solve/income` and `POST /solve/ages`

- Internal `withdrawalRate` derivation removed
- Response: `withdrawalRate` field removed from `/solve/income` response
