# v0.10 — Scenario editing and wizard state persistence

_Status: Complete_

---

## What's new

### 1. Wizard state persistence (no data loss on navigation)

Currently, navigating back from Assets → PersonDetails or clicking "Edit details" from the scenario screen discards all entered data by resetting `people` to `[]`. This iteration makes all wizard navigation non-destructive.

**`App.jsx` changes:**
- `people` state is never cleared on navigation. It is only updated via `onPersonDetailsComplete` and `onAssetsComplete`.
- `handleEditDetails()` becomes `() => setStep(1)` — no `setPeople([])`.
- A new `handleEditAccounts()` callback is added: `() => setStep(2)`.

**`PersonDetails.jsx` changes:**
- Receives an `initialPeople` prop (the current `people` array from App).
- On mount, populates name/age/partner toggle from `initialPeople[0]` and `initialPeople[1]` if present.
- On completion, the callback returns only the names and ages; App merges this into the existing `people` state, preserving `accounts` and `statePension` for any person whose name has not changed. If a person is added (partner toggle switched on), they are appended with empty accounts. If a person is removed (partner toggle switched off), they are dropped.

**`Assets.jsx` changes:**
- Already receives `people` from App. Ensure it populates its local `accounts` and `statePensions` state from the incoming `people` prop on mount, so that going back from the scenario screen and re-entering step 2 restores previously entered account data.
- No visual changes needed.

**`ScenarioScreen.jsx` changes:**
- The header `[Edit details]` button calls `onEditDetails` (which no longer clears data).
- A new `[Edit accounts]` button calls `onEditAccounts` — placed alongside `[Edit details]` in the header.

---

### 2. Save scenario

A "Save" button/link in the scenario screen header. On click, downloads the current `people` state as a JSON file using the same format as `ui/src/scenarios/*.json`:

```json
{ "people": [...] }
```

Filename: `<name1>-<name2>.json` for two people (lowercased, spaces replaced with hyphens); `<name1>.json` for one person. Uses `Blob` + `URL.createObjectURL` + a transient `<a download>` element.

---

### 3. Load scenario from file

A "Load" button/link in the scenario screen header (alongside Save). On click, opens a hidden `<input type="file" accept=".json">` and reads the selected file. On successful parse of a valid `people` array, sets `people` in App state and remains on step 3 (scenario screen). Shows a brief inline error message if the file cannot be parsed or has an unexpected shape.

An additional "Load saved scenario" link is added to PersonDetails (Step 1) — this is the entry point when the user has not yet started the wizard. On click, opens the same file picker; on success, sets `people` and jumps to step 3.

**Validation:** the loaded JSON must have a `people` array where each entry has at minimum `name` (string), `currentAge` (number), and `accounts` (non-empty array). No schema library — a short set of explicit checks is sufficient.

---

### 4. Link `/scenarios` page from Step 1

Add a "View example scenarios →" link below the Continue button on PersonDetails. Navigates to `/scenarios` using React Router `<Link>`.

---

### 5. Panel 3 cards interactive

Clicking a Panel 3 scenario card loads its retirement ages and monthly income into the Panel 1 controls. Visually: cards get `cursor-pointer` and a hover border treatment. The card that matches the current Plan 1 settings is highlighted (already is — this remains).

No additional API calls are needed; the Panel 1 controls' `onChange` handlers already trigger a new `POST /run`.
