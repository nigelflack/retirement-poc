# v0.12 — Schedule field renaming + monthly amounts + simple screen step-change inputs

_Status: In progress_

---

## What's new

### 1. API field renaming and monthly amounts (breaking change)

The schedule field keys are renamed for clarity. `fromYear` was ambiguous — different tracks used different reference points. The new names are self-documenting.

| Field | Old key | New key |
|-------|---------|---------|
| `contributionSchedule[].fromYear` | `fromYear` | `fromYearsFromToday` |
| `incomeSchedule[].fromYear` | `fromYear` | `fromYearsFromRetirement` |
| `capitalEvents[].year` | `year` | `yearsFromToday` |

Amount fields are also converted from annual to monthly, positioning the JSON as the UI save/load format (consistent with all other monetary inputs):

| Field | Old key | New key |
|-------|---------|---------|
| `annualIncomeTarget` | annual £ | `monthlyIncomeTarget` — monthly £ |
| `contributionSchedule[].annualAmount` | annual £ | `monthlyAmount` — monthly £ |
| `incomeSchedule[].annualAmount` | annual £ | `monthlyAmount` — monthly £ |

`statePension.annualAmount` is unchanged — state pension is conventionally quoted annually and the Assets UI uses it annually.

The adapter (`resolveSchedules`) multiplies all monthly fields by 12 before building dense arrays. The engine always works in annual terms internally.

All scenario JSON files (both copies of `bob-alice.json` and `nigel-mimi.json`) are migrated to the new keys. The CLI (`retirement.py`) reads the renamed keys from input files — no prompt changes needed.

### 2. Simple screen — contribution step-change (per person)

Panel 1 gains an optional sub-field under each person's retirement age spinner. When filled in, it defines a single step-change in that person's contributions in the years leading up to their retirement.

**Interaction:**
- The section is collapsed by default behind a disclosure link: "▼ Add contribution step-changes". Clicking expands it inline.
- Two sub-fields per person: "Reduces to £X/mo" and "in the last Y years before retirement"
- Both fields must be filled for the step-change to take effect; if either is blank, no step-change is sent for that person
- Leaving the section collapsed is equivalent to today's behaviour (flat contributions)
- When the retirement age spinner changes, Y remains fixed (the step-year shifts automatically)
- If Y ≥ years to retirement, show inline validation: "Exceeds time to retirement — ignored"
- On load from JSON: if a person's `contributionSchedule` has exactly two entries, the section auto-expands and is back-filled: `reducesTo = entry[1].monthlyAmount`, `lastYears = retirementYear - entry[1].fromYearsFromToday`. If more than two entries, section stays collapsed (full schedule still passed through in payload).

**Payload mapping:**
Contribution step-change for a person with two filled sub-fields produces a `contributionSchedule` with two entries:
```json
[
  { "fromYearsFromToday": 0, "monthlyAmount": <full monthly> },
  { "fromYearsFromToday": <retirementYear - lastYears>, "monthlyAmount": <reducesTo> }
]
```
Where `<full monthly>` is the flat default derived from the person's accounts (summed `monthlyContribution`). `retirementYear` here is that person's individual years-from-today to retirement (`person.retirementAge - person.currentAge`).

### 3. Simple screen — income step-change (household)

Panel 1 gains an optional sub-field below the monthly income input. When filled in, it defines a single step-change in the household income target during retirement.

**Interaction:**
- The section is collapsed by default behind a disclosure link: "▼ Add income step-change". Clicking expands it inline.
- Two sub-fields: "Reduces to £X/mo" and "after year Y of retirement"
- Both fields must be filled for the step-change to take effect; if either is blank, no `incomeSchedule` is sent (flat income behaviour)
- `reducesTo` must be less than the main income field; if not, show inline validation: "Reduced income must be less than target income"
- On load from JSON: if `incomeSchedule` has exactly two entries, the section auto-expands and is back-filled: `reducesTo = entry[1].monthlyAmount`, `afterYear = entry[1].fromYearsFromRetirement`. If more than two entries, section stays collapsed (schedule still passed through in payload).

**Payload mapping:**
Income step-change produces an `incomeSchedule` with two entries:
```json
[
  { "fromYearsFromRetirement": 0, "monthlyAmount": <main income field> },
  { "fromYearsFromRetirement": <afterYear>, "monthlyAmount": <reducesTo> }
]
```
The top-level `monthlyIncomeTarget` is always set to the main income field, regardless of whether an `incomeSchedule` is also present.

### 4. Load as input shortcut

The existing `[Load]` button on the scenario screen already replaces `people` state from a JSON file. In v0.12 it additionally populates the new step-change sub-fields using the back-fill rules described above.

If the loaded file contains `capitalEvents`, they are included in the simulation payload unchanged — but there is no UI to view or edit them on this screen. They are silently passed through.

The `[Save]` button downloads the current `people` state as JSON. In v0.12 it also includes:
- Each person's `contributionSchedule` if the sub-fields are filled (two-entry form, monthly amounts)
- The household `incomeSchedule` if the sub-fields are filled (two-entry form, monthly amounts)
- Any `capitalEvents` that were loaded from file (passed-through unchanged)
- `monthlyIncomeTarget` derived from the main income field

## Out of scope / deferred

- Capital events UI inputs — no form fields for entering capital events; load-from-file pass-through only
- `contributionSchedule` or `incomeSchedule` with more than two entries via the UI — requires advanced screen (v0.13)
- Panel 2 and Panel 3 solve calls always use the flat income/contribution model — they do not incorporate step-changes even when the user has filled in step-change fields. This is a known simplification: the solve endpoints (`/solve/income`, `/solve/ages`) do not accept schedule inputs. The suggestions shown in Panel 2/3 are therefore based on flat assumptions and should be read as indicative only when step-changes are active.

## API changes

Key renames and amount unit change within existing schedule fields:
- `annualIncomeTarget` → `monthlyIncomeTarget` (monthly £; adapter multiplies by 12)
- `contributionSchedule[].fromYear` → `fromYearsFromToday`
- `contributionSchedule[].annualAmount` → `monthlyAmount` (monthly £; adapter multiplies by 12)
- `incomeSchedule[].fromYear` → `fromYearsFromRetirement`
- `incomeSchedule[].annualAmount` → `monthlyAmount` (monthly £; adapter multiplies by 12)
- `capitalEvents[].year` → `yearsFromToday`

`statePension.annualAmount` is unchanged.

`POST /simulate` request/response shape is otherwise unchanged. The adapter multiplies all monthly fields by 12 before building dense arrays.
