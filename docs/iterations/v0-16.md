# v0.16 — Cashflow simulation engine

_Status: Complete_

---

## Goal

Rebuild the Monte Carlo simulation engine around a cashflow-first, multi-pot model. The engine receives a fully-resolved array of year items — one per simulation year from today to the household's `toAge` — and runs the MC simulation against them. All domain logic lives in the adapter (the routes layer): resolving income/expense schedules into year items, routing surplus between pots, honouring accessibility windows, managing draw priority.

**Scope: engine and CLI only.** The UI will break and is explicitly not addressed here.

---

## What's new

- **New engine** (`run.js` rewrite): single unified loop — no accumulation/drawdown phase split. One pass from year 0 to `toAge`.
- **Multi-pot tracking**: each pot has its own type and independent stochastic return distribution. Three types: `investments`, `property`, `cash`.
- **Year-array input contract**: each year item contains `income`, `expense`, `capitalOut`, `capitalIn`, `surplusOrder`, `drawOrder`.
- **New adapter** (`routes/run.js` rewrite): resolves the high-level scenario JSON into the year array for the engine.
- **New high-level API request format**: structured around pots, people, income/expense schedules, and routing strategy (see below).
- **Scenario JSON files updated** to new format (used by CLI for validation).
- **CLI formatter updated** for new response structure.
- **`simulation.json` updated** with per-asset-type return parameters.

---

## What's out of scope / deferred

- **Mortgage / liability pots** → backlog. Property appears in the year array as cashflows only (deposit as `capitalOut`, sale proceeds as `capitalIn`). Net equity is not correctly modelled — known limitation.
- **Tax modelling** (pension withdrawal tax, income tax bands, TFLS) → backlog.
- **Annual allowance enforcement** (pension £60k, ISA £20k) → the adapter uses `annualCap` values in `surplusStrategy` but does not validate them against HMRC limits. Caller's responsibility. Documented simplification.
- **Partial pot liquidation** (e.g. sell half the BTL) → backlog. Only full liquidation (`capitalIn` with no `amount`) is supported in v0.16.
- **Cross-asset return correlation** → backlog. Pot returns are sampled independently. This understates tail risk when equity and property fall together. Documented simplification.
- **Solve endpoints** (`/solve/income`, `/solve/ages`) → will 404. Deferred to a future iteration once the adapter is stable.
- **UI** → will break. Addressed in v0.17.

## Engine input contract

This is what `runFull()` receives. It is produced entirely by the adapter — never sent directly by the UI or CLI.

### Top-level structure

```json
{
  "pots": [ ... ],
  "primaryPot": "isa_nigel",
  "retirementYear": 10,
  "years": [ ... ]
}
```

- `primaryPot` — the pot ID that receives net cashflow (income − expense) each year before `surplusOrder` redistribution. Must be a liquid pot.
- `retirementYear` — year index of household retirement (earliest across all people). Used only for the accumulation snapshot in the response. The engine loop does not change behaviour at this boundary.

### Pot definition

```json
{ "id": "pension_nigel", "type": "investments", "initialValue": 250000 }
```

- `type` — one of `investments`, `property`, `cash`. Return parameters are looked up from `simulation.json` by type.
- `initialValue` — today's money. Not inflated — it is the actual current value.
- `property` type pots are **illiquid** — they never appear in `drawOrder` and are not counted in the solvency check.

### Year item

All `amount` values are in today's money. The engine inflates by `cumInflation` before applying.

```json
{
  "note": "Year 0 — both working",
  "income":     [ { "id": "nigel_salary", "amount": 75000 },
                  { "id": "mimi_salary",  "amount": 45000 } ],
  "expense":    [ { "id": "core_spending", "amount": 48000 } ],
  "capitalOut": [ { "id": "house_deposit", "toPot": "property", "amount": 200000 } ],
  "capitalIn":  [ { "id": "property_sale", "fromPot": "property" } ],
  "surplusOrder": [
    { "potId": "pension_nigel", "maxAmount": 40000 },
    { "potId": "pension_mimi",  "maxAmount": 20000 },
    { "potId": "isa_mimi",      "maxAmount": 20000 }
  ],
  "drawOrder": [ "isa_nigel", "isa_mimi", "pension_nigel", "pension_mimi", "cash" ]
}
```

- `note` — optional string. Engine ignores it; echoed in debug output.
- `income` — sources of household income this year. Summed and added to `primaryPot`.
- `expense` — household outgoings this year. Summed and subtracted from `primaryPot`.
- `capitalOut` — **mandatory** lump-sum transfers out of `primaryPot` into a target pot. Executed in order. Can cause ruin if `primaryPot` is insufficient (no clamping).
- `capitalIn` — lump-sum transfers into `primaryPot` from a source pot. If no `amount` is specified, the entire source pot is liquidated at its current stochastic value.
- `surplusOrder` — **adaptive** surplus routing. After mandatory transfers, any remaining positive `primaryPot` balance is distributed to these pots in order, up to `maxAmount` each, stopping when `primaryPot` hits zero.
- `drawOrder` — if `primaryPot` is still negative after the above, draw from these pots in order until covered or all exhausted (→ ruin).

### Engine sequence per year (per simulation path)

```
1. Sample inflation factor → cumInflation *= inflFactor
2. netCF = (sum income − sum expense) × cumInflation
   primaryPot += netCF
3. Execute capitalOut in order:
     primaryPot -= item.amount × cumInflation
     toPot      += item.amount × cumInflation
4. Execute capitalIn in order:
     amount = item.amount != null ? item.amount × cumInflation : fromPot value
     primaryPot += amount
     fromPot    -= amount (set to 0 for full liquidation)
5. Distribute surplus through surplusOrder:
     for each entry while primaryPot > 0:
       transfer = min(primaryPot, entry.maxAmount × cumInflation)
       primaryPot  -= transfer
       entry.pot   += transfer
6. If primaryPot < 0, draw through drawOrder:
     for each potId while primaryPot < 0:
       transfer = min(abs(primaryPot), pot value)
       primaryPot += transfer
       pot        -= transfer
     if primaryPot still < 0 → ruin (mark path, continue to next year for net worth tracking)
7. Apply stochastic return to every pot independently:
     pot *= exp(muLn + sigmaLn × sampleNormal())
8. Record sum(liquid pots) / cumInflation and each pot value / cumInflation for fan chart
```

**Post-ruin behaviour**: the path is marked as ruined but the loop continues to record net worth. This allows "ruined at 82 but owns a house worth £X" to be visible in debug output.

**Solvency**: sum of all pots where `type !== 'property'` ≥ 0.

## High-level API request (adapter input)

This is what POST `/simulate` accepts. The adapter resolves this into the engine's year array.

```json
{
  "label": "Base case",
  "pots": [
    { "id": "pension_nigel", "type": "investments", "owner": "nigel", "accessFromAge": 57, "initialValue": 250000 },
    { "id": "pension_mimi",  "type": "investments", "owner": "mimi",  "accessFromAge": 57, "initialValue": 40000  },
    { "id": "isa_nigel",     "type": "investments", "owner": "nigel", "accessFromAge": 0,  "initialValue": 61000  },
    { "id": "isa_mimi",      "type": "investments", "owner": "mimi",  "accessFromAge": 0,  "initialValue": 50000  },
    { "id": "cash",          "type": "cash",                           "accessFromAge": 0,  "initialValue": 30000  }
  ],
  "primaryPot": "isa_nigel",
  "people": [
    { "id": "nigel", "name": "Nigel", "currentAge": 52, "retirementAge": 62,
      "statePension": { "annualAmount": 12000, "fromAge": 67 } },
    { "id": "mimi",  "name": "Mimi",  "currentAge": 46, "retirementAge": 56,
      "statePension": { "annualAmount": 10000, "fromAge": 67 } }
  ],
  "incomeSchedule": [
    { "id": "nigel_salary", "annualAmount": 75000, "fromYear": 0, "toYear": 10 },
    { "id": "mimi_salary",  "annualAmount": 45000, "fromYear": 0, "toYear":  4 },
    { "id": "btl_rental",   "annualAmount":  7200, "fromYear": 0, "toYear": 18 }
  ],
  "expenseSchedule": [
    { "id": "core_spending",      "annualAmount": 60000, "fromYear": 0               },
    { "id": "reduced_retirement", "annualAmount": 36000, "fromYear": 20              }
  ],
  "capitalEvents": [
    { "id": "tequila_wharf_sale", "year": 18, "toPot": "isa_nigel", "amount": 200000 }
  ],
  "surplusStrategy": [
    { "potId": "pension_nigel", "annualCap": 48000 },
    { "potId": "pension_mimi",  "annualCap": 1200  },
    { "potId": "isa_nigel",     "annualCap": 20000 },
    { "potId": "isa_mimi",      "annualCap": 20000 }
  ],
  "drawStrategy": [ "isa_nigel", "isa_mimi", "pension_nigel", "pension_mimi", "cash" ],
  "toAge": 100,
  "debug": false
}
```

### Adapter rules

- **Year index convention**: year 0 = first simulation year (person age = `currentAge + 1` at end of year). `fromYear` is inclusive, `toYear` is exclusive.
- **Income schedule**: `statePension` entries are synthesised by the adapter per person, active from `fromAge - currentAge` onward.
- **Expense schedule**: if `fromYear` values overlap, later entries override earlier ones for the same `id`. Non-overlapping entries are additive.
- **Capital events** with `amount` → `capitalOut` if going to a non-primary pot; `capitalIn` entry added to primary pot otherwise. Capital events with no `toPot` land in `primaryPot` directly as net cashflow (adapter adds `amount` to income in that year).
- **Surplus strategy**: pension pots are excluded from `surplusOrder` in years where the owning person has retired (`year >= person.retirementAge - person.currentAge`). ISA and cash pots are always eligible while they have capacity.
- **Draw strategy**: a pot is included in `drawOrder` only if `year >= pot.accessFromAge - ownerCurrentAge`. Property type pots are never included.
- **Primary pot**: must be a liquid `investments` type pot.
- **Retirement year**: `min over people of (person.retirementAge - person.currentAge)`. Passed to engine as `retirementYear`.

## `simulation.json` changes

```json
{
  "numSimulations": 10000,
  "inflation": { "mean": 0.025, "stdDev": 0.005 },
  "returns": {
    "investments": { "mean": 0.07, "stdDev": 0.12 },
    "property":    { "mean": 0.03, "stdDev": 0.08 },
    "cash":        { "mean": 0.04, "stdDev": 0.0  }
  }
}
```

Cash return `stdDev: 0.0` means a fixed annual return (no stochastic sampling needed — the engine can special-case this).

## Response format

Top-level structure is unchanged from v0.15. Key notes:

- `portfolioPercentiles.byAge` — now represents sum of all liquid pots (investments + cash) at each simulation year.
- `accumulationSnapshot` — pot sum at `retirementYear` (from engine metadata), not a hard phase boundary.
- `survivalTable` — unchanged semantics (solvency at each drawdown year).
- `probabilityOfRuin` — unchanged.
- `debug: true` adds `resolvedYears` — the full year array as received by the engine, with each item's `nominalNetCF` added (the inflated net cashflow for that simulation year, averaged across paths). This replaces `resolvedSchedules` from v0.15.

## Decisions to record in `decisions.md`

1. **Cashflow-first engine with year-array input** — income/expense/capital items in today's money; engine inflates per year.
2. **Multi-pot with independent return distributions** — investments, property, cash types; cross-asset correlation not modelled (documents known simplification).
3. **Adapter holds all domain logic** — engine has no knowledge of retirement ages, pension rules, allowances, or accessibility windows. These are pre-resolved by the adapter into the year array.
4. **Surplus routing via `surplusOrder`, draw via `drawOrder`** — both per-year and adapter-resolved. Pension pots excluded from surplus routing after person retires; excluded from draw order before `accessFromAge`.
5. **Ruin continues (post-ruin net worth tracking)** — paths are marked ruined but the loop continues. Enables "ruined but house-rich" visibility.
6. **Semantic IDs not UUIDs** — item and pot IDs are adapter-generated semantic strings. Human-readable in debug output and CLI.
