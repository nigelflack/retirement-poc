# v0.17 — Simulation engine modularisation and naming

_Status: Complete_

---

## Goal

Refactor the Monte Carlo engine and its supporting math module to separate concerns cleanly: statistical model configuration, return/inflation sampling, cashflow mechanics, and results assembly. Also correct naming throughout — files, functions, and utilities — to remove ambiguity and reflect what things actually are.

No functional change to outputs. This is a structural improvement that makes each responsibility independently testable and creates the right seams for future enhancements (per-year varying assumptions, pluggable return models, deterministic testing via injectable RNG).

**Scope: server simulation layer only.** No API contract changes, no adapter changes, no CLI changes, no UI changes.

---

## What's new

### Pass 0 — naming

**File renames:**
- `simulation/run.js` → `simulation/engine.js` — unambiguous; `run.js` is used twice in the project and describes an action rather than a thing
- `routes/run.js` → `routes/simulate.js` — mirrors the endpoint it serves (`POST /simulate`)
- `simulation/run.test.js` → `simulation/engine.test.js` — follows the file rename

**Function renames:**
- `runFull` → `simulate` — `runFull` is a leftover from the old two-stage API era
- `sampleNormal` → `sampleStandardNormal` — describes what it returns, not how it's implemented
- `logNormalParams` → `lognormalFromArithmetic` — clarifies the direction of conversion
- `percentilesOf5` → `summaryPercentiles` — "of5" is cryptic (five percentiles, not the 5th)

**Removal:**
- `interpolateSolventAt` removed from `math.js` — stale since v0.16; references `e.age` which no longer exists on survival table entries; no callers. Will be rewritten correctly when solve endpoints are rebuilt.

**`server/src/index.js` updated** to require `routes/simulate.js` instead of `routes/run.js`.

### Pass 1 — seams and model extraction

**1. Move `summaryPercentiles` and `allPercentiles` to `math.js`**  
Pure functions with no engine dependency; should be available to any consumer of `math.js`.

**2. Extract `buildReturnModel(config)` → opaque model object**  
The `lognormalFromArithmetic` pre-processing currently runs inside `simulate`. Extract into `buildReturnModel(config)` in `math.js`. Makes the boundary between "configure" and "run" explicit.

**3. Extract `sampleInflationFactor(model, year, rng)` and `sampleReturnFactor(model, type, year, rng)`**  
The two inline sampling expressions become named functions. The `year` parameter is accepted but ignored today — future CMA models can use it to vary assumptions by year without any engine loop changes. Live in `math.js`.

**4. Injectable RNG (default `sampleStandardNormal`)**  
`simulate` accepts an optional `rng` parameter (defaults to `sampleStandardNormal`). Sample functions receive it. Enables deterministic tests without large `numSimulations`.

### Pass 2 — structural decomposition

**5. Extract `runPath(potDefs, years, retirementYear, model, rng)` → single simulation path**  
The inner simulation loop becomes a standalone function returning `{ ruined, retirementValue, retirementRealValue, liquidByYear, realByYear }`. `simulate` becomes: initialise accumulators → for each sim, call `runPath`, accumulate → assemble results.

**6. Extract named cashflow step functions**  
Steps 2–6 of the year loop become named functions:

- `applyNetCashflow(pots, yr, primaryIdx, cumInflation)`
- `applyCapitalTransfers(pots, yr, primaryIdx, potIndex, cumInflation)`
- `applySurplus(pots, yr, primaryIdx, potIndex, cumInflation)`
- `applyDraw(pots, yr, primaryIdx, potIndex)`
- `checkRuin(pots, isLiquid)` → `{ ruined, liquidTotal }`

Each is a pure mutation of the `pots` array, independently testable.

## What's out of scope / deferred

- Per-year varying CMA assumptions — the `year` parameter on sample functions creates the seam; the model stays flat for now
- Regime-switching or correlated return models — deferred to a future iteration
- Any API, adapter, CLI, or UI changes
